# 忠实性规则参考 (Faithfulness Rules)

> 本文档是 atlas subskill 的忠实性规则权威来源。SKILL.md 在生成过程中引用本规则，确保输出不编造、不推测未标注、不遗漏。

---

## 1. 三级置信度分类

所有语义陈述必须归入以下三类之一，不得模糊处理：

| 类别 | 标签 | 含义 | 使用时机 |
|------|------|------|----------|
| **代码明确实现** | （直接陈述，无需额外标签） | 代码中可以直接确认的行为——函数体中有对应语句、变量声明中有对应类型、调用链中有对应 `call` | 看到函数体、看到赋值语句、看到 `return`、看到 `if/else` 分支 |
| **推断自命名/注释** | `推测` | 变量名、函数名、注释暗示某种意图，但代码不完全证明 | 变量名 `_integral` 暗示积分、注释 `// clamp output` 但看不到 clamp 逻辑、函数名 `validate` 但看不到校验内容 |
| **无法从当前片段判断** | `无法判断` | 需要其他文件、运行时上下文或完整函数体才能确认 | 只有函数声明没有定义、依赖外部模块行为、`#include` 引入的类型无法解析 |

### 标注格式

- 表格中：在对应单元格写入 `推测` 或 `无法判断`
- Mermaid 图中：在节点文字末尾追加 `[推测]` 或 `[无法判断]`
- 伪代码中：在步骤末尾用括号标注 `（推测）` 或 `（无法判断）`
- Dashboard 中：置信度字段填写 `高` / `中` / `低`

### 判定优先级

```
看到函数体 → 归入"代码明确实现"
只看到声明/签名 → 归入"无法判断"
看到命名/注释但无函数体 → 归入"推测"
```

## 2. 绝对禁止规则

以下行为在任何模式下均不可接受，包括默认模式和 `--strict` 模式：

### 2.1 禁止把最佳实践当作已实现代码

```
❌ "代码使用了输入验证"       — 除非看到 if/else 或 throw 语句
❌ "代码处理了边界条件"       — 除非看到对应的条件分支
❌ "代码遵循了 RAII 模式"     — 除非看到构造/析构中的资源管理
✅ "代码中未观察到输入验证逻辑"
```

**判断方法**：搜索函数体中是否有对应的条件判断、异常抛出、返回错误码。如果没有，不得声称"已处理"。

### 2.2 禁止把注释意图当作运行时行为

```
❌ 注释写 "// 线程安全" → 不能声称代码是线程安全的
❌ 注释写 "// 处理所有错误" → 不能声称错误处理完备
❌ 注释写 "/* 计算 PID 输出 */" → 不能确认计算逻辑的正确性
✅ 注释写 "// 线程安全" → 可标注"注释声称线程安全[推测]"
```

**规则**：注释是意图声明，不是行为证据。只有可执行的代码语句才是行为证据。

### 2.3 禁止为了图表完整性而编造流程

```
❌ 流程图中添加代码未出现的分支路径
❌ 调用图中添加代码未出现的函数调用
❌ 数据流图中添加代码未出现的数据转换
❌ 状态机图中添加代码未出现的状态转换
```

**替代做法**：
- 在图中省略不确定部分
- 在图下方的文字说明中补充"当前片段未观察到 XXX 流程"
- 用 `[推测]` 标注推断的节点/边

### 2.4 禁止省略简单函数

```
❌ 跳过 getter（如 getKp()）
❌ 跳过 setter（如 setKp()）
❌ 跳过 reset/wrapper/adapter 函数
❌ 以"过于简单"为由不在表格中列出
```

**规则**：所有提取到的函数必须出现在函数契约总表和函数调用图中。getter/setter 可以在调用图中聚合在子图中，但函数名仍要列出。副作用矩阵中，纯读取函数的副作用列标记为"否"即可。

### 2.5 禁止生成泛泛夸奖式代码解释

```
❌ "代码结构清晰，设计良好"
❌ "代码遵循了良好的面向对象设计原则"
❌ "这个模块实现很健壮"
✅ "状态变量集中在 `_integral`, `_prevError`, `_output` 三个成员中，修改函数为 compute() 和 reset()"
```

**规则**：每个结论必须绑定具体代码证据。不带证据的评价性陈述只允许出现在最终审查结论（Section 13）的 bullet 列表中，且必须是可操作的建议。

## 3. 图表忠实性规则

### 3.1 Mermaid 图通用规则

| 规则 | 说明 |
|------|------|
| 只放代码确认的调用关系 | 调用图中只画 `source → target` 在代码中可确认的边 |
| 看不到函数体时不能画完整流程图 | 核心执行流程图：只有看到函数体才能画分支和步骤；否则输出"当前片段只包含函数声明，无法生成完整流程图" |
| 不确定依赖要标注 | 模块框架图中不确定的依赖加 `[推测]` 后缀 |
| 状态变量必须在代码中有声明 | 状态机图中的每个状态必须在代码中有对应的变量/枚举声明 |
| 节点名称简洁 | 不使用过长句子作为节点文字 |
| 不得为完整而编造 | 不得添加代码中不存在的节点或边 |

### 3.2 各图表类型的忠实性要求

#### 模块框架图 (flowchart LR)

- **必须包含**：当前文件/类（中心节点）、代码中可见的输入（函数参数）、代码中可见的输出（返回值/I/O）、代码中声明的状态变量
- **不得脑补**：看不到的外部依赖标 `[推测]`，完全未知的依赖不画

#### 核心执行流程图 (flowchart TD)

- **必须包含**（当函数体可见时）：分支 (`if/else`, `switch`)、状态更新节点、返回节点
- **不得脑补**：看不到函数体 → 输出"无法生成"声明，不画假设流程
- **缺失处理**：错误路径未确认时标注 `[未确认]`

#### 数据流图 (flowchart LR)

- **必须包含**：代码中可见的输入参数、声明的状态变量、代码中的计算步骤、返回值
- **中间计算节点**：只在代码中观察到对应计算语句时添加

#### 状态机图 (stateDiagram-v2)

- **触发条件**：存在 `enabled/disabled`、`initialized/running/closed`、`connected/disconnected`、`mode/state/status`、`error/failed` 等离散状态变量
- **必须映射**：每个状态对应代码中的变量/枚举声明，每个转换对应代码中的函数调用或条件语句
- **无状态机时**：输出"当前文件没有明确的离散状态机"

#### 函数调用图 (flowchart TD)

- **所有函数必须出现**：包括 getter/setter
- **调用关系必须真实**：只画代码中可确认的 `functionA → functionB` 调用
- **可分组**：getter/setter 可聚合在子图中，但函数名仍要列出
- **外部调用标注**：对未知外部函数标注 `external`

## 4. 覆盖率规则

覆盖率矩阵（Section 11）必须检查以下覆盖项。缺失元素标为 `未覆盖`，不得从矩阵中省略。

### 4.1 必须覆盖的检查项

| 检查项 | 要求 | 缺失时标注 |
|--------|------|------------|
| 函数覆盖 | 所有函数必须出现在**函数契约总表** | `未覆盖` |
| 状态覆盖 | 所有状态变量必须出现在**状态变量表** | `未覆盖` |
| 状态修改覆盖 | 修改状态的函数必须出现在**副作用矩阵** | `未覆盖` |
| 核心流程覆盖 | 核心函数必须有**流程图**或声明无法生成 | `部分覆盖` |
| 数据流覆盖 | 核心输入、状态、中间量、输出必须出现在**数据流图** | `部分覆盖` |
| 状态机覆盖 | 存在状态变量时必须有**状态机图** | `部分覆盖` |
| 调用覆盖 | 所有函数尽量进入**调用图** | `未覆盖` |
| 风险覆盖 | 高风险边界条件必须进入**风险矩阵** | `未覆盖` |

### 4.2 覆盖率矩阵的判定规则

| 规则 | 判定 |
|------|------|
| 函数未进入函数契约总表 | `未覆盖` |
| 状态变量未进入状态变量表 | `未覆盖` |
| 修改状态的函数未进入副作用矩阵 | `未覆盖` |
| 核心函数无流程图 | `部分覆盖` |
| 核心数据无数据流图 | `部分覆盖` |
| 有状态切换但无状态机图 | `部分覆盖` |
| getter/setter 进入函数表和调用图即可 | `完整覆盖` |

### 4.3 覆盖率自检流程

覆盖率矩阵生成后，必须执行以下自检：

1. 统计提取到的函数总数 → 对比函数契约总表中的函数数量
2. 统计识别到的状态变量总数 → 对比状态变量表中的变量数量
3. 检查每个修改状态的函数是否出现在副作用矩阵中
4. 检查每个核心函数是否有对应的流程图（或"无法生成"声明）
5. 任何缺失元素必须在覆盖率矩阵中标为 `未覆盖`

## 5. 严格模式 (`--strict`) 额外规则

`--strict` 模式在默认规则基础上追加以下约束：

### 5.1 标注强化

| 规则 | 行为 |
|------|------|
| 所有未确认行为 | 必须标注 `无法判断`，不得留空或使用模糊措辞 |
| 所有推测内容 | 必须显式标注 `推测`，不得以确定语气陈述 |
| 置信度字段 | 每个表格行和每个图表节点必须包含置信度标注 |

### 5.2 图表限制

| 规则 | 行为 |
|------|------|
| 图表内容 | 只能包含代码明确支持的信息 |
| 推测节点 | 在 `--strict` 模式下不得出现在 Mermaid 图中（改为文字说明） |
| 不确定调用 | 调用图中不得画推测的调用关系 |

### 5.3 风险建议绑定

| 规则 | 行为 |
|------|------|
| 风险建议 | 必须和代码证据绑定，格式：`风险点 → 代码位置 → 建议动作` |
| 不允许泛化 | 不得输出无证据的"最佳实践"建议 |
| 禁止无证据安全声明 | 不得使用"安全"、"健壮"、"可靠"等词汇，除非有对应代码证据 |

### 5.4 默认模式 vs 严格模式对比

| 维度 | 默认模式 | `--strict` 模式 |
|------|----------|-----------------|
| 推测内容 | 标注 `推测` 后允许出现 | 标注 `推测` 且不得出现在图表中 |
| 未确认行为 | 建议标注 | 必须标注 `无法判断` |
| 图表中的推测节点 | 允许（带 `[推测]` 标签） | 不允许，改为文字说明 |
| 风险建议 | 尽量绑定代码证据 | 必须绑定代码证据 |
| 泛化建议 | 允许但不推荐 | 完全禁止 |

## 6. 忠实性自检清单

生成 atlas 输出后，执行以下自检。任何一项不通过都必须修正后重新输出：

- [ ] **无脑补调用**：调用图中每条边在代码中有对应 `call` 语句
- [ ] **无注释当事实**：注释意图未作为确定行为出现在任何表格或图表中
- [ ] **无省略函数**：getter/setter/wrapper 均出现在函数契约总表和调用图中
- [ ] **无夸大安全**：未出现无证据的"安全"/"健壮"/"正确"描述
- [ ] **推测已标注**：所有非代码确认的内容标注了 `推测` 或 `无法判断`
- [ ] **覆盖率完整**：覆盖率矩阵中无遗漏的函数或状态变量
- [ ] **风险有证据**：风险矩阵中每行都有对应的代码位置
- [ ] **流程图真实**：核心执行流程图的每个节点对应代码中的实际语句
- [ ] **状态机有据**：状态机图的每个状态和转换有对应的代码声明/语句
- [ ] **副作用不遗漏**：修改状态的函数全部出现在副作用矩阵中

## 7. 冲突解决优先级

当规则之间产生冲突时，按以下优先级解决：

```
1. 绝对禁止规则（Section 2）          — 最高，不可违反
2. 三级置信度分类（Section 1）         — 分类不确定内容
3. 图表忠实性规则（Section 3）         — 限制图表内容
4. 覆盖率规则（Section 4）             — 确保不遗漏
5. 严格模式额外规则（Section 5）       — 仅 --strict 时追加
```

示例冲突：

- **覆盖率要求所有函数有流程图** vs **看不到函数体不能画流程图** → 优先遵守忠实性，输出"无法生成"声明，覆盖率矩阵标为 `部分覆盖`
- **调用图要求所有函数出现** vs **调用关系不确定** → 函数节点必须出现，但不确定的调用边不画（或在默认模式下标注 `[推测]`）
